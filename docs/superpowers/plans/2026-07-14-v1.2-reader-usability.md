# Novel Forge v1.2 Reader Usability Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the shared first-pass reader-kit workflow with isolated experiments, guided kit creation, previewed CSV mapping, conflict-safe evidence merging, conservative metrics, and reproducible CSV/XLSX reports.

**Architecture:** Each experiment owns its configuration, sample, questionnaires, responses, reports, and source hashes under `reader-kits/RE-NNN/`. Imports are parsed into an ephemeral preview, mapped to the canonical schema, validated, and merged only after explicit conflict decisions; the original accepted evidence remains untouched until one guarded transaction succeeds.

**Tech Stack:** TypeScript, TypeBox/YAML, existing CSV parser extended for mapping, ExcelJS, Node crypto/fs, wizard registry, existing transaction/guidance system.

## Global Constraints

- Complete the wizard-foundation plan first.
- Keep v1.1 `reader-experiments.yaml` and `reader-kit/` projects readable before migration.
- Migration must preserve every response field, metric, verdict, timestamp, source statement, and experiment identifier exactly.
- Only accepted `source: human` responses count in metrics or claims.
- CSV import previews before mutation and merges rather than replacing an experiment.
- Duplicate/conflicting reader-phase pairs require an explicit keep/use/exclude decision.
- Delayed evidence requires a matching accepted immediate response for the same reader.
- Claims remain bounded by sample size, segment, pairing, questionnaire version, and test design.
- Every create, migrate, and import operation is one guarded transaction and Git checkpoint with refreshed guidance.
