# Novel Forge 1.4-4 Premise and Story-Engine Laboratory Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Compare several recognizable structural versions of the writer's seed before book architecture hardens, while keeping selection under explicit writer control.

**Architecture:** Extend the 1.4 domain with a strict book-level `premise-lab.yaml` contract and pure validation/selection services. Add a state-neutral `premise-update` event and a loopback-only premise wizard that previews a complete comparison before saving it and records selection only by appending a writer decision in the canonical decision ledger. Book-plan prompts receive only the selected premise; drafting context never reads premise-lab variant prose.

**Tech Stack:** TypeScript 5.9, TypeBox/YAML, existing intake decision ledger, structured rejection envelope, guarded transactions, loopback wizard, static browser assets, Node test runner, Node 22.19.0 and Node 24 GitHub Actions matrix.

## Global Constraints

- Keep package version, lockfile, and dependencies unchanged.
- A premise lab contains three to five variants.
- Variant order is exactly 1 through N. Order 1 is the raw-idea baseline or closest faithful expansion.
- Every variant preserves every declared top-level seed element.
- Every normalized story engine is unique.
- Variant diagnostics are neutral observations. The schema contains no score, rank, winner, recommendation, or model-selection field.
- `selected_variant_id` and `selection_decision_id` are both null or both non-null.
- A selection decision must be an unreplaced writer decision with matching book scope, subject `premise-selection`, and choice equal to the selected variant ID.
- No automatic selection, score-based selection, or selection without a decision ID.
- New books receive an empty premise-lab template. Existing approved 1.3 books without the file remain valid.
- Newly rebuilt book plans block when a premise-lab file exists but has no valid selected variant.
- `premise-update` is state-neutral, book-planning-only, and may write only the active book premise lab and `series/decision-ledger.yaml`.
- The premise wizard snapshot and prompt must not read or expose taste-profile influence references.
- The book-plan prompt receives only the selected variant. Nonselected premise text never enters chapter drafting context.
- The browser previews before apply; stale proposals use the existing structured rejection envelope.
- No manuscript, project/book state, gates, reader evidence, publishing, marketing, package output, or guidance writes through premise-update.
- Use TDD and pass the full Node 22.19.0 / Node 24 matrix before merge.
