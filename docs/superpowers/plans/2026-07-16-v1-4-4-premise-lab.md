# Novel Forge 1.4-4 Premise and Story-Engine Laboratory Implementation Plan

> **For agentic workers:** Use `superpowers:subagent-driven-development` or `superpowers:executing-plans`. Complete tasks in order and keep every writer decision explicit.

**Goal:** Compare several recognizable structural versions of the writer's seed before book architecture hardens, while keeping selection under explicit writer control.

**Architecture:** Add a strict book-level `premise-lab.yaml`, pure validation and selection services, a state-neutral `premise-update` event, and a loopback-only premise wizard. Book-plan prompts receive only the selected premise. Chapter drafting context never reads premise-lab prose.

**Tech stack:** TypeScript 5.9, TypeBox/YAML, the existing decision ledger, structured rejections, guarded transactions, static local browser assets, Node tests, Node 22.19.0, and Node 24.

## Global constraints

- Keep the package version, lockfile, and dependencies unchanged.
- A populated premise lab contains three to five variants.
- Variant order is exactly 1 through N.
- Order 1 is the raw-idea baseline or closest faithful expansion.
- Every variant preserves every declared top-level seed element.
- Every normalized story engine is unique.
- Diagnostics are neutral observations. The schema contains no score, rank, winner, recommendation, or model-selection field.
- `selected_variant_id` and `selection_decision_id` are both null or both non-null.
- A valid selection requires one unreplaced writer decision with matching book scope, subject `premise-selection`, and choice equal to the selected `PV-NNN` ID.
- No automatic, score-based, or model-selected premise is allowed.
- New books receive an empty premise-lab template.
- Existing approved 1.3 books without the file remain valid.
- A rebuilt book plan blocks when a populated premise lab has no valid selection.
- `premise-update` is state-neutral, book-planning-only, and may write only the active premise lab and `series/decision-ledger.yaml`.
- Premise generation and snapshots do not read or expose taste-profile influence references.
- Book planning receives only the selected variant and structural consequences.
- Chapter drafting receives no premise-lab prose.
- Browser actions preview before apply and use existing structured stale-state rejections.
- `premise-update` cannot mutate manuscript, project/book state, gates, reader evidence, publishing, marketing, packaging, `STATUS.md`, or `HANDOFF.md` directly.

**Files**

- `src/domain/v1-4-premise-schemas.ts`

**Required interfaces**

```ts
export function defaultPremiseLab(bookId: string): PremiseLab;
export function premiseLabFindings(lab: PremiseLab, ledger?: DecisionLedger | null): V14Finding[];
export function selectPremise(lab: PremiseLab, ledger: DecisionLedger, variantId: string, decisionId: string): PremiseLab;
export function selectedPremiseContext(lab: PremiseLab | null): string;
export function premiseComparison(lab: PremiseLab): Array<{ id: string; engine: string; observations: string[] }>;
```

### Test requirements

Prove:

- fewer than three or more than five variants fail;
- duplicate IDs, orders, or normalized story engines fail;
- order must be contiguous;
- exactly one raw-idea baseline exists and it is order 1;
- raw idea and seed elements are required once variants exist;
- every variant preserves all declared seed elements;
- blank premise, question, consequence, differentiation, or series-potential fields fail;
- score, rank, winner, and recommendation fields are rejected by the strict schema;
- `selected_variant_id` and `selection_decision_id` must be paired;
- an unknown selected variant fails;
- selection without the exact active writer decision fails;
- a valid decision-bound selection passes;
- selected context contains only the chosen variant and its structural consequences;
- comparison output contains engines and neutral observations but no winner claim.

Normalize story engines using Unicode normalization, lowercase, punctuation collapse, and whitespace collapse. Compare seed elements case-insensitively. Validation returns all blockers and never mutates input.
