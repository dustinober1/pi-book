# Novel Forge v1.2 — Existing-Project and Reader Usability Design

**Status:** Conversation design approved; pending written-spec review  
**Version target:** 1.2.0  
**Repository:** `dustinober1/pi-book`  
**Branch:** `agent/v1.2-existing-project-reader-usability`

## 1. Purpose

Novel Forge 1.1 introduced first-pass support for manuscript adoption, reader kits, packaging readiness, and adding a subsequent book. Version 1.2 turns those primitives into complete guided workflows for authors who already have manuscript material, need disciplined outside-reader evidence, want production-ready publishing and marketing exports, and are continuing a series.

The normal entrypoint remains `/novel`. The extension may open a temporary localhost browser wizard for workflows that are materially easier to review visually. The browser is a review and proposal surface only. All accepted changes must pass through Novel Forge's existing application services, schemas, guarded transaction engine, derived guidance generation, Git checkpoints, and undo workflow.

## 2. Product goals

Version 1.2 must deliver four outcomes:

1. **Guided adoption of an existing project** with high-fidelity DOCX and EPUB support, chapter and asset preview, author-controlled mapping, and non-destructive import.
2. **Reader-kit generation and evidence import** with experiment-isolated files, merge-safe CSV handling, conflict review, conservative metrics, and spreadsheet summaries.
3. **A packaging checklist wizard** that explains and routes every incomplete prerequisite, then produces a complete author, publishing, audiobook, and marketing package.
4. **A context-aware next-book workflow** that previews inherited series facts and unresolved threads, gathers the new book's author decisions, and records provenance without inventing canon.

## 3. Non-goals

Version 1.2 will not:

- create a persistent dashboard, hosted service, desktop application, or cloud account;
- allow the browser to write directly to project files;
- silently resolve manuscript ambiguity, reader-response conflicts, metadata gaps, or series continuity decisions;
- claim reader validation beyond the accepted human sample;
- preserve arbitrary source-program layout with pixel-perfect fidelity;
- replace professional cover design, accessibility review, proofreading, legal review, or retailer submission;
- make generated marketing copy canonical until the author approves it.

## 4. Chosen architecture

### 4.1 Embedded temporary browser wizard

`/novel` starts an ephemeral HTTP server bundled with the Pi extension and binds it to `127.0.0.1` only. The extension generates a cryptographically random, unguessable per-session token, opens the user's default browser, and serves bundled static assets plus a narrowly scoped project API.

The server stops when:

- the Pi session ends;
- the author chooses **Close wizard**;
- the session exceeds its idle timeout; or
- the underlying project becomes unavailable.

The wizard is not externally hosted and has no analytics, remote scripts, remote fonts, or third-party assets.

Source files enter the session only through a Pi-native path prompt or a browser file upload copied into a session-owned temporary directory. The browser cannot submit an arbitrary server-side path for reading.

### 4.2 Read/propose/apply boundary

The browser may:

- read a sanitized project snapshot;
- request previews and conversion diagnostics;
- stage chapter, asset, response, metadata, checklist, and inheritance decisions in ephemeral session state;
- submit a typed proposal for confirmation.

The browser may not write to the project filesystem or invoke arbitrary commands.

Every accepted proposal follows this path:

```text
Browser wizard
  -> token-authenticated local session controller
  -> validated typed application request
  -> Novel Forge application service
  -> schema and stage validation
  -> guarded transaction
  -> primary and derived file writes
  -> Git checkpoint
```

This preserves:

- allowlisted paths;
- expected-stage and project-state-hash checks;
- atomic rollback;
- explicit author confirmations;
- `STATUS.md` and `HANDOFF.md` regeneration;
- approval evidence hashes;
- one event per Git checkpoint; and
- undo through `git revert`.

### 4.3 Conversion strategy

Pandoc is the preferred conversion engine when available. It must be detected at runtime rather than required at installation. Calls use argument arrays, fixed timeouts, controlled working directories, captured stdout/stderr, and no shell interpolation.

A bundled Node fallback keeps adoption and packaging usable without Pandoc. The fallback must preserve readable structure and embedded assets, but it may report reduced fidelity for layout constructs it cannot reproduce safely.

Every conversion report identifies:

- engine and version used;
- source and output hashes;
- warnings and unsupported constructs;
- extracted or omitted media;
- fidelity limitations; and
- whether the result is suitable for review or final production.

## 5. Workflow 1 — Existing-project adoption

### 5.1 Entry and supported sources

When the active book has no manuscript chapters, `/novel` offers **Adopt an existing project** as a normal guided action. It remains available under Advanced tools as well.

Supported inputs:

- `.docx` manuscript;
- `.epub` manuscript;
- a single `.md` or `.txt` manuscript;
- a directory of `.md` or `.txt` chapter files.

The source must remain unchanged.

### 5.2 Discovery pass

Before any project mutation, Novel Forge creates a temporary adoption preview containing:

- detected front matter, chapter sections, interludes, appendices, and back matter;
- source order and proposed manuscript order;
- proposed chapter numbers, titles, slugs, and destination paths;
- source and proposed word counts;
- headings, scene breaks, emphasis, quotations, links, footnotes, and notes;
- extracted images, captions, alt text when present, and approximate placement;
- source metadata such as title, author, language, identifiers, and description, marked unverified;
- tracked changes, comments, equations, tables, floating objects, text boxes, unsupported fonts, or other fidelity warnings.

High-fidelity preservation means preserving meaningful document structure and visual intent where safe: heading hierarchy, paragraph styles, emphasis, block quotations, lists, tables, footnotes, scene breaks, ornaments, links, chapter art, illustrations, maps, and image captions. Fonts, margins, page furniture, and unsupported decorative layout may be normalized into maintainable Markdown and asset references.

### 5.3 Browser mapping controls

The author can review and change the proposal before import:

- reorder sections;
- rename a chapter or section;
- change chapter numbering;
- split a section at a detected heading or explicit marker;
- combine adjacent sections;
- classify content as front matter, manuscript chapter, interlude, appendix, or back matter;
- exclude content;
- change image placement or caption;
- mark discovered metadata as accepted, edited, or ignored.

The wizard must prevent duplicate destination paths and duplicate accepted chapter numbers.

### 5.4 Asset handling

Embedded images are extracted to the active book's assets directory using deterministic, collision-safe filenames. The importer preserves captions and approximate placement in Markdown. It records original names, source relationships, hashes, dimensions, media types, and final destinations.

The importer rejects path traversal, executable content, remote-resource loading, unsafe archive entries, and decompression-bomb patterns. It imposes configurable limits on compressed size, expanded size, archive entry count, media count, image dimensions, and chapter count.

### 5.5 Atomic adoption

Confirmation submits one adoption proposal to the application layer. A successful transaction writes:

- normalized manuscript chapters;
- front and back matter files when present;
- extracted assets;
- `adoption-map.yaml`;
- `adoption-report.md`;
- accepted but unverified metadata candidates;
- updated `BOOK.yaml` counts and state;
- regenerated `STATUS.md` and `HANDOFF.md`.

The transaction creates one Git checkpoint. Any validation or write failure rolls back the entire adoption.

### 5.6 Post-adoption guidance

Novel Forge must not treat an imported manuscript as though it had passed the normal planning, review, or approval pipeline. After adoption, `/novel` identifies missing evidence and recommends the correct next action, including voice capture, book-plan reconstruction, continuity review, canon review, reader testing, publishing metadata, or packaging.

No plot, canon, approval, reader evidence, or series decision may be invented during adoption.

## 6. Workflow 2 — Reader kits and evidence import

### 6.1 Experiment isolation

Each experiment has its own canonical directory:

```text
books/book-01/reader-kits/RE-001/
  experiment.yaml
  sample.md
  immediate-questions.md
  delayed-questions.md
  responses.csv
  import-report.md
  summary.md
```

The book-level experiment index may summarize experiments, but experiment state and source files live with the experiment.

Version 1.1 layouts using `reader-kit/` and `reader-experiments.yaml` remain readable. A guarded migration reorganizes them without altering recorded responses, metrics, verdicts, timestamps, or source claims.

### 6.2 Kit creation wizard

The wizard guides the author through:

- sample scope: first page, first chapter, selected chapters, act, excerpt, or full manuscript;
- exact target-reader segment;
- minimum immediate and delayed response counts;
- delayed follow-up interval;
- blind or labeled test design;
- optional A/B variant naming;
- questionnaire version;
- sample preview and word count;
- privacy and handling instructions.

The generated kit includes immediate and delayed questionnaires, instructions that prevent coaching and sample reopening, and a response CSV with the exact schema version.

### 6.3 CSV preview and mapping

CSV import never applies immediately. The browser first shows:

- detected schema version;
- row and reader counts;
- immediate/delayed coverage;
- validation errors;
- unmapped or extra columns;
- duplicate reader-phase pairs;
- conflicts with accepted responses;
- delayed rows without matching immediate rows;
- non-human or unsupported sources;
- questionnaire-version mismatches;
- experiment-ID mismatches.

For a compatible but differently named CSV, the author may map source columns to canonical fields. The mapped shape must validate before import.

### 6.4 Merge semantics

Imports merge accepted rows into existing evidence rather than replacing the entire experiment.

For each conflict, the author must choose one of:

- keep the existing accepted response;
- use the imported response;
- exclude the response from accepted evidence.

Every conflict decision is recorded in `import-report.md`. Existing accepted evidence remains intact unless the author explicitly replaces or excludes it.

Novel Forge rejects simulated, model-generated, or otherwise non-human sources from reader metrics. Such material may be stored only as clearly labeled non-evidence notes outside accepted response collections.

### 6.5 Metrics and summaries

Metrics are recomputed only from accepted human rows. Verdicts remain conservative and cannot exceed the evidence supported by target segment, response count, pairing quality, and experiment design.

The generated summary covers:

- immediate and delayed coverage;
- continuation and purchase intent;
- delayed hook and signature-moment recall;
- talkability and recommendation specificity;
- repeated confusions and trust breaks;
- segment and variant differences;
- contradictory responses;
- experiment limitations;
- supported conclusions;
- unsupported conclusions and prohibited claims.

A generated XLSX and CSV package provides the same evidence in tabular form for filtering and review.

## 7. Workflow 3 — Packaging checklist wizard

### 7.1 Guided checklist

At packaging stage, `/novel` opens a browser checklist rather than only displaying a pass/fail block. Each checklist item shows:

- status;
- blocking or advisory classification;
- why the item matters;
- evidence paths;
- exact repair action;
- whether a repair can be started from the wizard;
- the source hash associated with any approved or generated output.

The browser action returns a typed request to Novel Forge; it does not repair or write directly.

### 7.2 Checklist coverage

The checklist covers at least:

- contiguous, compilable manuscript;
- manuscript approval;
- canon lock;
- blocking revision tickets;
- reader-evidence claim limits;
- publishing metadata completeness;
- marketing metadata completeness and approval state;
- cover, illustration, map, ornament, and other asset readiness;
- copyright, language, territorial, edition, and publication fields;
- ISBN placeholders or assigned identifiers by format;
- retailer descriptions;
- keywords and categories;
- audiobook metadata;
- accessibility and alt-text notes;
- DOCX, EPUB, Markdown, CSV, and XLSX export readiness.

Incomplete blocking items prevent final package approval. Novel Forge does not fill missing metadata with invented values.

### 7.3 Canonical metadata

Each book gains two versioned canonical files:

```text
books/book-01/publishing.yaml
books/book-01/marketing.yaml
```

`publishing.yaml` contains structured production and retailer metadata, including:

- title, subtitle, series name, and series number;
- author and contributors;
- language and territory;
- copyright and publication dates;
- edition and format information;
- ISBN placeholders or values by format;
- the author-selected or approved retailer description used for export;
- keywords and categories;
- trim, formatting, accessibility, and asset notes;
- audiobook narrator, producer, duration placeholder, and distribution metadata.

`marketing.yaml` contains draft and approved promotional assets, including:

- positioning statement;
- target audiences;
- hooks and selling points;
- retailer-copy variants from which the publishing description may be selected;
- launch copy;
- social posts;
- ad variants;
- audiobook promotion;
- series-page copy;
- approval status and notes for each asset group.

Generated outputs are reproducible derivatives. They never become the canonical metadata source.

### 7.4 Complete author package

The packaging workflow generates, as applicable:

- manuscript DOCX;
- EPUB;
- clean Markdown archive;
- normalized asset bundle;
- publishing metadata XLSX and CSV;
- reader-response XLSX and CSV;
- retailer descriptions;
- keyword and category recommendations;
- launch announcement and launch-plan copy;
- social posts;
- advertising variants;
- audiobook metadata and promotional copy;
- series-page copy;
- packaging report;
- conversion reports.

Generated marketing text remains draft material until explicitly approved in `marketing.yaml`. Category recommendations are also drafts and must be checked against the current retailer taxonomy before submission.

### 7.5 Staging and failure behavior

Exports are produced in a temporary staging directory. Final outputs move into place only after required manuscript and metadata exports succeed and validate.

A failed marketing-text generation does not invalidate successful manuscript exports. A failed required manuscript conversion prevents final package completion.

Existing approved outputs are not silently overwritten. If source hashes change, the wizard shows stale outputs and requires explicit regeneration.

## 8. Workflow 4 — Context-aware next book

### 8.1 Eligibility and proposal

Once the active book is canon-locked or packaged, `/novel` proposes the next available `book-NN`. Forced creation remains an Advanced action with explicit confirmation and provenance warnings.

The proposal previews:

- series identity and positioning;
- continuing characters;
- unresolved threads and promises;
- established world, institutional, technology, relationship, and timeline facts;
- facts that must not change;
- previous profile and target length;
- reader findings that may inform but not dictate the next book.

### 8.2 Author decisions

The wizard asks the author to confirm or provide:

- title or working title;
- role in the series;
- relationship to the prior book: direct continuation, adjacent story, prequel, later installment, or other;
- profile;
- target length;
- protagonist or primary viewpoint;
- unresolved threads allowed to continue;
- threads intentionally deferred;
- immutable facts;
- excluded context that should not be inherited.

### 8.3 Created files and provenance

Creation writes the next book's standard structured files plus:

```text
books/book-02/inheritance-report.md
books/book-02/inherited-context.yaml
```

The inherited context distinguishes:

- accepted inherited canon;
- unresolved continuing threads;
- optional context;
- explicitly excluded material;
- author decisions still required.

No new plot solution, character outcome, reader result, or canon fact may be invented. The operation is one guarded transaction and Git checkpoint, followed by refreshed guidance recommending the next planning action.

## 9. Security and safety requirements

### 9.1 Wizard session

- Bind only to `127.0.0.1`.
- Require the random per-session token on every API request and invalidate it when the session closes or expires.
- Enforce same-origin requests and reject unexpected origins.
- Expire idle and closed sessions.
- Keep all staged proposals ephemeral until confirmation.
- Permit access only to the current project and source files selected through an approved selection flow.
- Serve no remote executable content.

### 9.2 Archive and document parsing

- Reject unsafe archive paths and symlinks.
- Disable external XML entities and remote URL resolution.
- Limit compressed and expanded sizes, entry counts, media counts, image dimensions, and chapter counts.
- Reject encrypted, corrupted, empty, or unsupported sources with actionable diagnostics.
- Preserve source files read-only.

### 9.3 External commands

- Use argument arrays rather than shell command strings.
- Use controlled working and output directories.
- Apply execution timeouts.
- Capture engine versions and logs.
- Never pass untrusted values through shell interpolation.

## 10. Compatibility and migration

Version 1.2 uses project metadata version `1.2.0`.

- New projects write 1.2-compatible metadata and directories.
- Version 1.1 projects remain readable without immediate migration.
- `/novel` warns when a project can benefit from the v1.2 migration.
- Migration is explicit, guarded, reversible through Git, and preserves recorded evidence.
- Missing `publishing.yaml` or `marketing.yaml` is an actionable warning, not a parsing failure.
- Projects written by a newer installed Novel Forge version remain blocked as in 1.1.

## 12. Acceptance criteria

Version 1.2 is complete when:

1. `/novel` can launch a token-protected temporary browser wizard bound only to loopback.
2. A DOCX or EPUB can be previewed, remapped, and adopted without modifying the source, with images and meaningful formatting preserved and all writes applied atomically.
3. A project without Pandoc can complete the supported workflows through the Node fallback and receive accurate fidelity warnings.
4. Reader experiments are isolated by experiment, compatible v1.1 evidence can migrate without semantic changes, and CSV imports merge only after preview and explicit conflict decisions.
5. Reader metrics and claims use accepted human evidence only.
6. The packaging wizard routes incomplete items to exact repairs and generates the complete production and marketing package from canonical metadata.
7. Publishing and marketing outputs are reproducible from versioned YAML and manuscript sources.
8. The next-book wizard previews inherited context, collects author decisions, records provenance, and invents no new canon.
9. All accepted browser actions pass through guarded transactions, produce Git checkpoints, and refresh `STATUS.md` and `HANDOFF.md`.
10. The full Node 22.19.0 and Node 24 CI matrix passes.

## 13. Release boundary

The implementation will be delivered on a dedicated branch and opened as a reviewable pull request. It remains unmerged unless the author explicitly requests a merge. The `v1.2.0` tag must not be created until the PR is merged and the resulting `main` commit passes the full CI matrix.
