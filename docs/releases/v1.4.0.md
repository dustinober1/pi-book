# Novel Forge v1.4.0 — Author Velocity

Novel Forge 1.4 improves guided authoring speed and recoverability without weakening writer control or evidence boundaries.

## Added

- Brief bootstrap that records the source brief as read-only evidence and converts only explicit or visibly inferred setup details into typed intake records.
- A neutral premise laboratory that compares structural variants while leaving the final premise choice to the writer.
- Resumable persistent runs with explicit targets, pause, resume, cancellation, stale-state protection, and bounded chapter work.
- Rolling chapter-packet windows that preserve completed work and refill only the next useful planning horizon.
- Structured guarded-event rejection codes with one corrected retry for schema or reference payloads and mandatory reload for stale state.
- Author-journey evaluations covering brief-to-plan, rolling drafting, pause/resume, and bounded revision work.
- Runtime profiles for full, local, and tiny-local environments, with deterministic work, context, graph, ticket, and prompt limits.

## Writer-control boundaries

- Writer approval remains mandatory at every existing human gate.
- The system does not select the final premise for the writer.
- Persistent or automated runs stop at approval gates, blockers, continuity conflicts, reveal-order conflicts, missing research, and invalid context.
- Public-review observations remain market evidence and never become human reader evidence for the current manuscript.

## Compatibility

- Existing 1.3 projects remain readable.
- Older project metadata is upgraded explicitly; an upgrade does not invent evidence, change approvals, or rewrite manuscript prose.
- Existing accepted manuscripts, decisions, reader experiments, canon, and story threads remain authoritative.

## Evaluation boundaries

This workflow does not guarantee literary quality, publication success, reader validation, or a specific wall-clock speed. The release evidence verifies deterministic workflow behavior, safety boundaries, compatibility, packaging, and test coverage.

## Release artifact

The guarded release workflow verifies the exact merged commit on Node 22.19.0 and Node 24 before creating annotated tag and GitHub release `v1.4.0`.
