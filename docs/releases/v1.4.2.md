# Novel Forge v1.4.2 — Repository organization

Novel Forge v1.4.2 adds a guarded way to turn a half-complete, mixed writing repository into an in-place Novel Forge project without silently promoting notes into story truth.

## Install

```bash
pi install git:github.com/dustinober1/pi-book@v1.4.2
```

For one temporary Pi session:

```bash
pi -e git:github.com/dustinober1/pi-book@v1.4.2
```

Use a copy or verified backup for the first organization run.

## Repository organization

Run a read-only preview from the uninitialized repository root:

```text
/novel-organize --dry-run
```

Then apply with explicit project metadata:

```text
/novel-organize My Novel --profile thriller --type standalone --target-words 100000
```

The organizer:

- scans mixed chapters, drafts, outlines, notes, research, documents, and supported assets;
- excludes Git internals, existing archives, symlinks, ignored/generated/dependency files, code, configuration, lockfiles, and repository documentation;
- rejects nested Git repositories, stale or modified previews, canonical state collisions, and pre-existing staged work;
- treats explicit chapter headings and numeric filenames as structural evidence while keeping name/folder classifications provisional;
- copies confirmed bytes into conservative manuscript or source-material paths;
- verifies source and canonical SHA-256 hashes before moving originals;
- moves confirmed superseded sources into `.archive/<timestamp>/files/`;
- records original, canonical, and archive paths, hashes, sizes, reasons, and confidence in `manifest.yaml`;
- rolls back live failures and uses literal, path-only Git checkpoints that exclude unrelated work.

The author separately confirms the proposed structure, acknowledges provisional classifications, and approves the exact archive list. An imported partial manuscript remains at voice intake/book planning.

## Writer-control boundaries

Writer approval remains mandatory at every human gate. Repository organization does not approve voice, plans, chapters, reviews, canon, or packaging.

The system does not select a writer's premise. Premise comparison and selection remain explicit author decisions.

Authorized brief bootstrap remains read-only with respect to the supplied brief and keeps inference visibly separate from writer decisions.

Imported notes never become canon, plot state, ready research claims, review findings, real-reader evidence, or approvals merely because the organizer classified or copied them.

## Compatibility

Existing 1.3 and 1.4 projects remain readable. `/novel-organize` is only for an uninitialized repository; existing Novel Forge projects continue to use adoption, migration, integrity, and metadata-upgrade workflows.

Tags `v1.4.0` and `v1.4.1` remain immutable.

## Verification boundaries

The release qualification covers type checks, unit and integration tests, deterministic evaluations, package import and clean-start checks, release-tree verification, and package contents. Passing these checks demonstrates workflow and safety contracts; it is not proof of literary quality, full-book model performance, publication success, or real-reader validation.
